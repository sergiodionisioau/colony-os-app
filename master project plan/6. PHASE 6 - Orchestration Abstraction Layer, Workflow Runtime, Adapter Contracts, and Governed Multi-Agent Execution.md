19. Zero-tolerance baseline

These are non-negotiable.

19.1 Workflow baseline
no workflow without schema
no workflow without version
no workflow without dependency validation
no workflow without declared tool/data requirements
no risky workflow branch without explicit approval behavior
19.2 Execution baseline
no run without actor identity
no run without adapter compatibility validation
no step without persisted state
no state transition without audit evidence
no long-running governed workflow without checkpoint support where required
no hidden runtime side effects outside artifact/state model
19.3 Adapter baseline
no adapter without contract conformance
no framework-specific bypass of control plane
no silent unsupported feature degradation
no incompatible workflow allowed to start if incompatibility is known
no adapter state that cannot normalize back into platform state
19.4 Agent baseline
no child agent without bounded identity
no child agent without supervision
no orphan child agent on parent cancel
no self-escalation through orchestration framework
no cross-workspace or cross-run bleed
19.5 Tool and data baseline
no tool execution outside allowlist
no retrieval outside approved profiles
no secret access outside declared policy
no undeclared module or connector access
no hidden persistence outside Phase 5 data plane
19.6 Recovery baseline
no retry without policy
no retry on unsafe non-idempotent step without compensation path
no cancel without child cleanup behavior
no pause/resume without state integrity
no failure hidden from operator view
19.7 UX baseline
operator must understand what is running
operator must understand which step failed
operator must understand which approvals are blocking
operator must understand which adapter is in use
operator must be able to pause, resume, cancel, or kill within policy bounds
20. Happy path

This is the required success scenario for Phase 6.

20.1 Workflow execution happy path
User opens workflow catalog
User selects a validated workflow
User sees compatible adapters and chooses the workspace default
System validates input, policy, tool/data requirements, and adapter compatibility
Workflow run is created
Tasks dispatch in dependency order
Artifacts pass between steps through governed storage
Workflow completes successfully
Run summary and evidence are available
20.2 Multi-agent happy path
A supervisory workflow reaches an agent-spawn step
System creates bounded child agent identities
Child agents receive permission profiles and scoped tool/data access
Child agents perform assigned subtasks
Outputs normalize into artifact records
Parent workflow collects results and continues
Supervisor can terminate a child if policy or timeout requires
Final run completes with full audit chain
20.3 Approval-gated happy path
Workflow reaches a sensitive branch
Execution pauses at approval gate
Human reviewer sees context and risk summary
Reviewer approves
Workflow resumes from paused state
Downstream steps execute
Approval decision is preserved in evidence
20.4 Pause/resume happy path
Operator pauses a long-running workflow
Active steps drain or checkpoint
Run enters paused state
Operator resumes later
Resume validation passes
Workflow continues from checkpoint
Final outputs remain intact
20.5 Failure and retry happy path
A step fails with a retryable error
System records failure classification
Retry policy allows another attempt
Retry runs with backoff
Step succeeds
Workflow completes
Evidence shows original failure and successful retry
20.6 Cancellation happy path
Operator cancels a running workflow
No new steps are dispatched
Child agents are terminated or drained according to policy
Artifacts and state are preserved
Run ends in cancelled state
Operator can inspect partial outputs and evidence
20.7 Orchestrator swap happy path
Workspace admin changes default adapter
System validates compatibility across current published workflows
Incompatible workflows are flagged clearly
Selection change succeeds
Existing active runs remain on old adapter
New runs use new adapter
Audit trail records the change

If these seven paths do not work, Phase 6 is incomplete.

21. Failure conditions

Phase 6 fails if any of these occur:

a workflow can run without validation
a framework adapter can bypass policy, identity, or audit
a child agent can spawn without bounded supervision
tool or data access occurs outside declared requirements
workflow state is lost on pause/resume or failure recovery
retries happen on unsafe steps without compensation or idempotency
workflow cancellation leaves orphan activity
adapter swap silently breaks workflow compatibility
operator cannot explain what ran, what failed, or why it stopped
evidence for execution decisions is incomplete

23. Acceptance test for Phase 6

Ask these exact questions.

Workflow test

Can the platform validate and execute a workflow without depending on a single hard-coded orchestration framework?

Adapter test

Can each supported adapter prove it conforms to the platform execution contract?

Multi-agent test

Can the system spawn bounded child agents, supervise them, collect outputs, and terminate them cleanly?

Governance test

Can every workflow step prove it only used approved tools, data, memory, and permissions?

Recovery test

Can the platform pause, resume, retry, compensate, or cancel without losing control of state?

UX test

Can an operator explain what is running, what failed, what is waiting for approval, and what recovery options exist?

Swap test

Can a workspace change its default orchestrator without silently breaking active or future workflow behavior?

If any answer is no, Phase 6 is not done.

24. Recommended build order inside Phase 6

Use this exact order.

orchestrator registry and adapter schemas
workflow definition and validation
workflow run and step state models
task scheduler and execution runtime
adapter contracts and normalization
agent spawn and supervision
tool routing and approval gates
checkpoint, retry, timeout, compensation
pause/resume/cancel and recovery
module/data/control-plane integration
orchestration UX surfaces
hardening and test matrix

That order is optimal because the workflow model must exist before runtime, runtime before adapter normalization, and policy-safe recovery before premium UX.
