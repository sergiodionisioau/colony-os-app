18. Unit tests

These unit tests are mandatory.

18.1 Metric definition and formula tests
validate metric definition creation with valid unit and scope
reject metric definition missing formula or unit
verify derived metric uses only declared source metrics
verify metric version change does not silently overwrite prior version
verify metric with incompatible scope mix is rejected
18.2 Score computation tests
compute workflow reliability score from known inputs
compute score with missing optional inputs and reduced confidence
reject negative weights or invalid weight totals
verify score decomposition output is stable
verify score versioning preserves old observations
18.3 Telemetry ingestion tests
accept valid event
reject malformed event
reject event with cross-tenant scope mismatch
deduplicate duplicate event id or duplicate payload according to rules
record dropped event count on ingestion failure
handle late-arriving telemetry without corrupting prior windows
18.4 Workflow analytics tests
success rate calculation with zero denominator safe handling
p50/p95 latency calculation correctness
retry rate calculation correctness
cost per successful run when failures exist
version comparison correctness across equal windows
18.5 Evaluation and regression tests
evaluation run against fixed baseline
regression triggered only when threshold exceeded
no regression on statistically insignificant or policy-below-threshold delta
severity assignment correctness
benchmark with empty outputs handled explicitly
18.6 Alerting tests
threshold crossing creates alert
repeated threshold crossings dedupe correctly
resolution closes alert state
acknowledgement changes state correctly
alert suppression prevents noisy duplicate storms within configured window
18.7 Recommendation tests
recommendation generated only when evidence criteria met
recommendation ranking sorts by impact and risk correctly
recommendation requiring approval cannot auto-apply
stale evidence invalidates or downgrades recommendation
post-change evaluation attaches outcome back to recommendation
18.8 KPI and cost tests
KPI formula produces correct value
cost aggregation across runs produces expected totals
incompatible currency/provider-cost references rejected or normalized per policy
ROI calculation with zero benefit or zero cost handled explicitly
negative cost or impossible token counts rejected
18.9 Security and isolation tests
cross-tenant metrics query rejected
cross-workspace dashboard aggregation rejected unless explicitly allowed
hidden system-only metric not exposed to wrong role
recommendation evidence access scoped correctly
alert evidence access scoped correctly
18.10 UX/query consistency tests
dashboard time window matches drill-down time window
warning shown for partial data windows
scope label preserved across compare view
stale metrics banner shown when pipeline lag exists
missing evidence link surfaced as integrity error, not silent omission
19. Edge cases

These edge cases must be handled explicitly.

19.1 Data sparsity edge cases
zero workflow runs in window
one workflow run only
low sample count causing noisy regression signal
missing benchmark baseline
partial telemetry after outage
19.2 Time-window edge cases
overlapping windows with different granularity
daylight/timezone normalization issues
out-of-order telemetry arrival
clock skew between subsystems
delayed cost records arriving after workflow completion
19.3 Scope edge cases
workspace renamed but historical metrics preserved
module removed but historical analytics still queryable
adapter disabled but old runs still visible
archived environment included/excluded per policy
role loses access during active dashboard session
19.4 Calculation edge cases
division by zero in KPI or ROI
negative or impossible values
NaN/infinite intermediate math
missing submetric in weighted score
mismatched units across source metrics
19.5 Alerting edge cases
alert storm from repeated retries
threshold flapping just above/below boundary
duplicate alerts from same root cause across steps
acknowledged alert reopens on recurring breach
resolved alert evidence later updated
19.6 Recommendation edge cases
evidence stale before approval
recommendation conflicts with another queued recommendation
recommendation predicted improvement but post-change got worse
recommendation references deprecated metric version
high-confidence recommendation built on incomplete cost data
19.7 Governance edge cases
recommendation auto-generated for environment not allowed by policy
benchmark tries to compare incompatible workflow versions
score definition edited while dashboard query in flight
deleted module leaves orphan metric references
tenant suspension while alerts remain open
20. Zero-tolerance baseline

These are non-negotiable.

20.1 Metrics baseline
no metric without versioned definition
no score without decomposable formula
no dashboard tile without traceable source metrics
no silent scope mixing
no silent missing data substitution
20.2 Telemetry baseline
no telemetry event accepted without scope validation
no dropped telemetry hidden from operators
no out-of-order normalization that corrupts history silently
no unowned event source
no cost metric without traceable source record
20.3 Evaluation baseline
no regression claim without baseline reference
no benchmark without input and scoring definitions
no severity assignment without explicit thresholds
no benchmark result without evidence links
no incompatible version comparison without warning or block
20.4 Recommendation baseline
no recommendation without evidence
no recommendation without expected impact statement
no hidden auto-apply by default
no recommendation routed to live change without policy and approval path
no stale recommendation treated as current without revalidation
20.5 Alerting baseline
no alert without evidence and severity
no unresolved alert silently disappearing
no alert dedupe that hides materially different failures
no SLO reporting without defined window and scope
no error budget math without explicit budget source
20.6 Security baseline
no cross-tenant metrics leakage
no role-inappropriate access to analytics or evidence
no hidden system metrics exposed to standard users
no recommendation evidence leaking sensitive secrets or payloads
no audit gap for metric definition, alert, or recommendation changes
20.7 UX baseline
operator must know data freshness
operator must know score basis
operator must know why alert fired
operator must know why recommendation exists
operator must be able to trace summary to drill-down evidence
21. Happy path

This is the required success scenario for Phase 7.

21.1 Workspace health happy path
Operator opens workspace dashboard
System shows current health score, active alerts, workflow reliability trend, cost trend, and top recommendations
Operator clicks a degraded workflow
Workflow analytics screen shows latency, success rate, retries, failures, and cost by version
Evidence links explain why reliability dropped
21.2 Evaluation and regression happy path
Team publishes a new workflow version
Evaluation run is launched against benchmark inputs
System compares outputs to baseline version
Regression detection flags a latency regression but no quality regression
Regression record is created with severity and evidence
Release owner reviews result before rollout
21.3 Recommendation happy path
System detects high retry rate and elevated cost for one workflow
Recommendation engine proposes timeout and adapter configuration adjustment
Recommendation includes expected impact, risk class, and evidence
Admin reviews and approves recommendation
Proposal is routed into governed change lifecycle
After rollout, post-change metrics show improvement
Recommendation record closes with realized outcome
21.4 KPI and ROI happy path
Product owner binds “cost per resolved case” KPI to workflow and cost metrics
KPI dashboard shows weekly trend
New automation release reduces cost and increases throughput
ROI efficiency view shows before/after delta with evidence
Owner uses this to justify expansion
21.5 Alerting happy path
Adapter instability causes workflow failure spike
Alert rule triggers
Alert appears in alert center with severity, affected targets, and evidence
Operator acknowledges, investigates drill-down, and resolves after fix
Alert history remains preserved

If these five paths do not work, Phase 7 is incomplete.

24. Acceptance test for Phase 7

Ask these exact questions.

Metrics test

Can every metric and score shown in the UI be traced back to a versioned definition and source evidence?

Analytics test

Can an operator identify which workflow, agent, module, or adapter is degrading and why?

Evaluation test

Can the platform compare versions, detect regressions, and explain them with baseline evidence?

Cost/KPI test

Can the platform tie automation behavior to both cost and business outcomes credibly?

Recommendation test

Can the system generate optimization proposals with evidence and require approval before live change when policy requires?

Alerting test

Can alerts fire with clear severity and evidence, avoid pointless noise, and remain stateful through acknowledge/resolve cycles?

Security test

Can analytics and evidence stay correctly isolated by tenant, workspace, environment, and role?

If any answer is no, Phase 7 is not done.

25. Recommended build order inside Phase 7

Use this exact order.

metric and score definition registries

workflow/agent/module/adapter analytics
retrieval/memory/data quality analytics
cost and KPI bindings
evaluation and regression engine
alerting, SLOs, and anomaly baseline
recommendation engine and governance
dashboards and drill-down UX
audit and evidence packaging
unit tests and edge-case hardening
handoff to Phase 8

That order is optimal because definitions and telemetry must exist before analytics, analytics before evaluation, evaluation before recommendations, and all of that before polished UX.
